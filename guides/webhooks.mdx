## Overview

Instead of polling the API, you can register a webhook URL and ByVeya will
`POST` a JSON payload to it whenever an event occurs on your platform. This
is the recommended way to react to asynchronous operations like letter
printing, social post publication, or campaign completion.

## Supported events

| Event | Triggered when |
|-------|---------------|
| `letter.generated` | A letter PDF is successfully generated |
| `letter.printed` | A letter is confirmed as printed and dispatched |
| `sms.sent` | An SMS message is sent via Twilio |
| `email.sent` | A transactional email is sent via Resend |
| `social.post.queued` | A social post is accepted into the queue |
| `social.post.published` | A social post is successfully published to the network |
| `social.post.failed` | A social post fails to publish |
| `campaign.created` | A new email campaign is created |
| `campaign.completed` | An email campaign finishes sending |

## Registering a webhook

```bash
curl -X POST https://byveya.com/api/v1/webhooks \
  -H "x-api-key: bv_live_YOUR_KEY_HERE" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://myapp.com/webhooks/byveya",
    "events": ["sms.sent", "letter.generated"],
    "description": "Production webhook — SMS and letters",
    "secret": "my-signing-secret-keep-this-safe"
  }'
```

Leave `events` empty (`[]`) to subscribe to **all events**.

## Payload structure

Every webhook delivery is a `POST` request with the following JSON body:

```json
{
  "event": "sms.sent",
  "platform_id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "timestamp": "2026-03-23T12:00:00.000Z",
  "data": {
    "log_id": "abc123...",
    "to": "+447911123456",
    "twilio_sid": "SM1234567890abcdef",
    "status": "sent"
  }
}
```

## Verifying signatures

When you register a webhook with a `secret`, ByVeya includes an
`X-ByVeya-Signature` header with each delivery:

```
X-ByVeya-Signature: sha256=a1b2c3d4e5f6...
```

Verify it in your handler:

<CodeGroup>

```javascript Node.js
import { createHmac } from "crypto";

function verifySignature(rawBody, signature, secret) {
  const expected = "sha256=" + createHmac("sha256", secret)
    .update(rawBody)
    .digest("hex");
  return expected === signature;
}

// Express example
app.post("/webhooks/byveya", express.raw({ type: "application/json" }), (req, res) => {
  const sig = req.headers["x-byveya-signature"];
  if (!verifySignature(req.body, sig, process.env.BYVEYA_WEBHOOK_SECRET)) {
    return res.status(401).send("Invalid signature");
  }
  const event = JSON.parse(req.body);
  console.log("Received:", event.event);
  res.sendStatus(200);
});
```

```python Python
import hmac
import hashlib

def verify_signature(raw_body: bytes, signature: str, secret: str) -> bool:
    expected = "sha256=" + hmac.new(
        secret.encode(), raw_body, hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, signature)
```

```php PHP
function verifySignature(string $rawBody, string $signature, string $secret): bool {
    $expected = "sha256=" . hash_hmac("sha256", $rawBody, $secret);
    return hash_equals($expected, $signature);
}
```

</CodeGroup>

<Warning>
  Always verify signatures using a **constant-time comparison** to prevent
  timing attacks. The examples above use `hmac.compare_digest` (Python),
  `hash_equals` (PHP), and Node's `crypto.timingSafeEqual` equivalents.
</Warning>

## Responding to deliveries

Your endpoint must return a `2xx` status code within **10 seconds** to be
considered a successful delivery. If it doesn't, the delivery is marked as
failed and the `failure_count` on the webhook is incremented.

ByVeya does **not** automatically retry failed deliveries — you can inspect
them via `GET /api/v1/webhooks/:id` and reprocess if needed.

## Managing webhooks

```bash
# List all webhooks
GET /api/v1/webhooks

# Get details + last 20 deliveries
GET /api/v1/webhooks/:id

# Pause a webhook
PATCH /api/v1/webhooks/:id  { "is_active": false }

# Delete a webhook
DELETE /api/v1/webhooks/:id
```
